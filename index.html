01 — Opportunity Validation

Why this problem, and why now

Restaurant menus are static. The people eating from them are not. Menus fail three distinct types of diners in ways that a conversation can fix and a filter UI cannot.

⚠

Safety constraints

Diners with nut allergies, dairy intolerance, or religious dietary restrictions (no beef, halal) need certainty the menu cannot provide. Asking a waiter repeatedly feels burdensome and the answers are often incomplete.

?

Unfamiliar cuisines

Ordering from a cuisine you have never encountered means guessing what dishes contain, how they are pronounced, and whether they match your preferences. One wrong order can define the whole meal.

🔆

Preference matching

Mood, budget, appetite, and taste preferences are contextual and conversational. "Something light, not too spicy, around 400 rupees" is a natural request that no static menu filter can handle.

Diners with dietary restrictions, food preferences, or unfamiliarity with a cuisine struggle to navigate restaurant menus confidently. Static menus do not filter, explain, or personalise. For those with allergies, a wrong choice is not just disappointing, it is a safety risk. Asking a waiter feels socially uncomfortable. The result: people play it safe and miss out, or take risks they should not take.

V1 framing decision: diner-side PDF upload, not restaurant-deployed

The natural assumption was to build MenuMate as a restaurant-deployed tool. I rejected this. A restaurant deployment requires admin panels, multi-tenancy, onboarding, and a sales cycle before a single diner can use the product. V1 should validate the AI behaviour, not the distribution model. By framing V1 as a diner-side PDF upload tool, I could prove the core value (allergen safety, dish explanation, contextual recommendation) with zero dependency on restaurant adoption. This also gave me a shareable live demo for any restaurant menu, instantly. The B2B restaurant integration is V2 once the AI behaviour is validated.

02 — Design

Key product decisions

Every decision below was a genuine tradeoff. I have documented the options I considered and why I chose what I chose, because that is where the PM thinking lives, not in the output.

01
RAG pipeline over long-context approach
The simpler alternative was long-context: attach the full menu PDF to every API call and let the LLM read it directly, no vectors needed. For a single small menu, this would have worked. I chose RAG because cost per query stays flat regardless of menu size, multiple menus are scoped cleanly by menu ID, and retrieval focuses LLM attention on the 5 most relevant dishes rather than a full document where attention degrades in the middle. The honest trade-off: RAG added significant build complexity that long-context would have avoided. I accepted that trade-off in favour of an architecture that scales correctly and reflects how production AI products are actually built.
02
Allergen inference only at query time, never at ingestion
Gemini could have inferred allergens from dish names and descriptions during PDF extraction and stored them in the database. I explicitly prohibited this. If inferred allergen data looks identical to confirmed data in the database, no downstream system can distinguish between them. A future engineer, a future model, a future integration would have no way to know whether the allergens field contains restaurant-confirmed data or a guess. Inference belongs at query time, behind the allergen cascade's caveat structure, where the confidence level is always visible and honest.
03
Hybrid retrieval: SQL filter first, then semantic search
Safety-critical constraints (allergens, vegetarian flag, price) are extracted from natural language and converted to SQL filters before any vector search runs. A 0.87 semantic similarity score is not acceptable for a nut allergy exclusion. SQL gives a guaranteed yes/no; semantic search gives a probability. The allergen exclusion filter runs deterministically on the filtered pool. The semantic layer then ranks remaining dishes by intent. If semantic search ran first, it could fill the retrieval slots with dishes that fail hard constraints.
04
Dynamic temperature per query type, not one global setting
A single conservative temperature (0.2) would produce consistent allergen language but robotic recommendations. A single permissive temperature (0.4) would feel warm but risk inconsistent safety language. I set temperature dynamically based on query classification: allergen queries at 0.1 (safety language must be consistent), factual queries at 0.2 (one correct answer), recommendation queries at 0.4 (conversational warmth). The query classification step, a separate Gemini call that runs before response generation, serves double duty: it extracts SQL filters AND determines which temperature the response will be generated at.
05
System prompt structured around LLM attention distribution
LLMs pay more attention to the top and bottom of their context than the middle. This is called the "lost in the middle" problem. I structured the system prompt with safety-critical rules at the top (Core Rules, allergen cascade), functional rules in the middle, and all NEVER rules reinforced a second time at the bottom. The most critical allergen safety rule appears three times. This is not redundancy for its own sake. It is intentional placement in the highest-attention regions. The self-check block, a 9-point checklist the model runs before every response, sits at the very end as a final gate.
06
Evaluation framework designed before writing any code
I defined 35 test cases across 6 failure categories before the build began. This forced me to define what "correct behaviour" actually means for each scenario before I could be tempted to adjust the definition to match what the system produced. Without pre-defined pass/fail criteria, testing becomes confirmation bias. Failures have a clear fix category: system prompt, classification prompt, pipeline logic, or temperature. The eval framework is what turned a build exercise into a product delivery, with documented iteration and root cause analysis rather than informal "it seems to work" testing.

03 — Development

Build, debug, and ship

The build surfaced 20 documented issues across infrastructure, pipeline logic, and system prompt behaviour. Three of these are worth calling out specifically because they reveal something about how I think about product quality.

PM gap

Conversation memory was never specified as a requirement

The chat pipeline sent only the current message to Gemini on every call. Gemini is stateless. Without conversation history, the product could not maintain allergen constraints across turns, hold a warning under pushback, or connect a follow-up question to the dish just discussed. Four eval test failures traced to this single gap. The fix was architectural, not a prompt patch. The learning: conversation memory is a product requirement that maps to a specific API implementation. It was never written down.

PM gap

Stack locked before evaluating language compatibility

The embedding library specified at design (sentence-transformers) is Python-only. The runtime chosen (Supabase Edge Functions) runs Deno and TypeScript. These are incompatible. The switch was discovered mid-build, forcing a full embedding layer redesign. The learning: stack decisions are product decisions. Language compatibility, runtime limits, deployability, and cost must be evaluated together before the stack is locked, not discovered independently as each layer is built.

Shared

Ingestion pipeline hit Supabase CPU limits at 25 dishes

The original ingest-menu function ran PDF extraction and all dish embeddings inside a single serverless invocation. At 25 dishes it hit Supabase's 2-second CPU limit and failed. The fix was splitting into two functions: ingest-menu handles extraction, embed-dishes runs paginated embeddings in batches of 6 using the null embedding as a natural cursor. The learning: serverless functions have hard resource limits. Any pipeline with multiple sequential operations must be validated against those limits at realistic input sizes, not toy examples.

04 — Evaluation

Three rounds, one product that works

I defined 35 test cases across 6 failure categories before the build began. The eval ran in three rounds, with root cause analysis after each failure set before any fixes were applied.

Round Tests Pass rate Key fix applied Root cause insight
Run 1 35 54% Constraint parser rewritten; named dish SQL lookup added 5 of 12 failures traced to a single parser bug, not 5 independent issues
Run 2 13 retested 66% Multi-turn conversation history implemented Remaining failures shared one root cause: no conversation memory in the pipeline
Run 3 13 retested 100% System prompt additions for context resolution and constraint persistence All 35 tests passing including all safety-critical allergen and grounding tests
The most important eval insight: When multiple test failures appear, categorise by root cause before fixing anything. In Run 1, five failures that appeared to be independent system prompt issues were actually one constraint parser bug with five manifestations. One parsing prompt change fixed them all. The highest-leverage fix is always the one that addresses the most failures at the root cause level.
05 — Learnings

What I would do differently

These are not things that went wrong. They are decisions I would make earlier, with more information, if I were starting again.

1

Lock the stack at Stage 1, not Stage 4

I made stack decisions incrementally: language, then embedding library, then LLM provider, then runtime. Each decision constrained the next in ways I did not anticipate. Sentence-transformers was chosen before I knew the runtime was TypeScript. Gemini was chosen for cost before I evaluated its free tier rate limits against actual testing workload. A PM who makes stack decisions should evaluate all constraints simultaneously: language compatibility, runtime limits, deployability, demo format, and API reliability at testing pace. These are not independent choices.

2

Specify conversation memory as a product requirement before build

The product was supposed to remember stated allergies, maintain context across turns, and hold safety warnings under pushback. None of this was written as a requirement. A single-turn API implementation is the natural default when no multi-turn requirement is specified. The result was four eval failures that looked like system prompt problems but were actually a missing architectural requirement. "The assistant should remember what was discussed this session" is a product requirement that maps to a specific API implementation. It needed to be written down at Stage 3, not discovered at eval.

3

Test the system prompt, not just write it

The system prompt was reviewed for completeness but not tested against representative queries before the build began. Tone issues, intent misclassification, recommendation vs dish-query behaviour gaps, and categorical hallucinations from filtered context all surfaced in eval testing. A system prompt is a product artifact. It requires a test pass the same way code does. Writing "warm and helpful" in the tone section and assuming Gemini will deliver that without verification is not product work, it is wishful thinking.

06 — What is next

V2 direction

V1 validates the AI behaviour. V2 is about distribution and scale. These are the four directions that follow naturally from what V1 proved.

🏝

Restaurant-side integration

Deploy MenuMate as a QR-code accessible assistant on the restaurant's own menu. Shifts from diner-uploaded to restaurant-hosted. Unlocks a B2B revenue model and real-time menu updates.

📷

Scanner input for physical menus

Camera scan of a physical menu unlocks a diner-side tool that works at any restaurant without prior deployment. This is a potential pivot from B2B to B2C with a much larger addressable market.

👥

Group ordering mode

Handling multiple constraint sets at one table (one person has a nut allergy, another is vegetarian) is a strong AI edge case. The architecture already supports it. It was explicitly scoped out of V1 to contain complexity.

🌍

Multilingual support

Both personas have language dimensions: Anshul faces a language barrier, Priya may use Hinglish. Explicit input language detection and language-matched responses are a natural V2 feature, especially as the product scales beyond English-first markets.
